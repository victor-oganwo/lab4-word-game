# # First Impressions - Initial Take on the Project Assignment

## Initial Thoughts
At first, I saw the project as a simple Hangman-style word game, but as I started working, I understood that it was about more than just making the game run. The project also focused on planning, testing, documentation, and showing how AI was used during development. My first goal was to understand the structure of the assignment and build the game step by step instead of rushing everything at once.

## Assumptions Made
At the start, I assumed the main challenge would just be coding the game logic. I also thought CoPilot would mostly give short answers that matched the exact stage of the lab. As I continued, I realized I had to be more selective because some suggestions were useful, but others went further than what was needed at that moment.

## Points Needing Clarification
A few things were not fully clear to me at the beginning:
- how much of the project was expected before using CoPilot
- how detailed the notes and report needed to be
- how small or complete the “minimal core” should be
- how to keep the code simple while still separating logic from UI properly

# Key Learnings

## Computer Science Concepts and Technical Skills
One important thing I learned was how to break a project into smaller helper functions instead of trying to do everything inside one big block of code. I also learned more about separating game logic from user interaction, which made the code easier to manage. Another key learning was the importance of validating input, handling edge cases, and testing functions instead of just assuming they work.

## Insights about Using CoPilot Effectively
I learned that CoPilot was most useful when I asked focused questions about one step at a time. It worked better when I asked for help with a specific function, review, or explanation, instead of asking for a full solution. I also learned that using AI well is not about accepting everything it gives. It is more about checking whether the suggestion really fits the assignment.

## New Concepts or Tools Encountered
During the project, I worked more with pytest for testing and got more practice organizing a small Python project with files like `README.md`, `JOURNAL.md`, `MY_NOTES.md`, and `REPORT.md`. I also became more familiar with using different CoPilot modes for different tasks.

# Report on CoPilot Prompting Experience

### Types of prompts that worked well
The prompts that worked best were the ones that were short, direct, and limited to one task. For example, asking CoPilot to review a specific function, explain a specific suggestion, or help with the next helper function worked well. Asking for test ideas or documentation for one file was also useful. These kinds of prompts made the responses easier to control and evaluate.

### Types of prompts that did not work well or failed
Prompts that were too broad were less helpful because they sometimes led to answers that were bigger than what I needed for that stage of the lab. In some cases, the response was not exactly wrong, but it was more complex than necessary for the assignment. That meant I had to keep checking the lab instructions and simplify the suggestions before using them.

# Limitations, Hallucinations and Failures

## Examples of Hallucinations or Failures or Misleading Information or Confident but Wrong Explanations, or Over-engineered or Under-engineered Solutions
The main issue I noticed was that CoPilot sometimes tried to make the solution more complete than required. For example, when I only needed a minimal core function, it sometimes suggested adding extra state or expanding the design earlier than necessary. Some answers were useful for later parts of the project, but not always for the exact step I was working on.

## Analysis of Why These Issues Occurred
I think this happened because AI often tries to be generally helpful and fill in missing details, even when the task is meant to stay small. If the prompt was not narrow enough, it would sometimes assume I wanted a more advanced version of the solution.

## Impact on the Project
This did not stop me from finishing the project, but it meant I had to stay careful. Instead of copying everything directly, I had to compare the suggestion with the lab instructions and only keep the parts that actually matched what was required.

# AI Trust

## When did I trust the AI?
I trusted the AI more when I was using it for structure, documentation, reviewing a function, or suggesting tests. I also trusted it more when the answers were easy to compare directly with the lab requirements.

## When did I stop trusting it?
I became more careful when the answers started going beyond the current step of the assignment or adding more complexity than I needed. That was usually a sign that I needed to slow down and decide for myself what to keep.

## What signals or situations or patterns indicated low reliability?
The main signs were:
- answers that felt bigger than the question I asked
- suggestions that introduced extra features too early
- changes that made the code more complex than the lab required
- responses that sounded confident but still needed to be checked against the instructions

# What I Learned

## What did you learn about software development?
I learned that it is much better to build software step by step. Breaking the game into smaller functions made the project easier to understand and easier to improve. I also learned that testing, input validation, and clear separation of responsibilities are important even in a small project.

## What did you learn about using AI tools?
I learned that AI can be helpful, but only when it is used carefully. It is good for brainstorming, reviewing, documenting, and suggesting tests, but it should not replace my own judgment. I still need to understand the code and make the final decisions.

## When should you trust AI? When should you double-check it?
I should trust AI more for small, focused tasks and for giving ideas. I should double-check it when it starts making design decisions, expanding the scope, or giving answers that are more advanced than the current requirement.

# Reflection

## Did AI make you faster? Why or why not?
Yes, AI made me faster overall because it helped me move through the project without staying stuck on one part for too long. It was especially useful when I needed suggestions for the next step, for documenting code, or for thinking about tests. At the same time, it did not remove the need to think carefully, because I still had to decide what actually fit the assignment.

## Did you feel in control of the code?
Yes, most of the time. I felt more in control when I used AI for smaller tasks and checked the results before accepting them. That helped me keep the project aligned with the lab instructions.

## Would you use AI the same way next time? What would you change?
I would still use AI, but I would keep using it in a controlled way. I would continue asking smaller and more precise questions rather than broad ones. I would also keep treating AI as a tool for support rather than something to follow automatically.
